# Atlas Build Week — final demo recording plan

Target runtime: **2:45–2:55**. Hard stop: **3:00**.

## Before recording

- Record at 1920×1080 with the browser at 90–100% zoom.
- Use the public judge-demo URL once it is live.
- Close personal tabs, bookmarks, notifications and password-manager popups.
- Keep the browser, the repository page and a terminal with `npm test` ready.
- Record clean English narration. Do not add copyrighted music.
- Make one uninterrupted product-led take; trim only the first and last seconds.

## Shot-by-shot script

### 0:00–0:22 — Real problem, real proof

**Screen:** Atlas hero and the 14-month trajectory.

**Narration:**

> In May 2025, Bacchus Holiday Homes started with no brand, no digital visitor journey and no connected destination system. Fourteen months later, Project Atlas connects three real accommodations and direct booking with roughly 240 curated places, a regional Explorer, map, Live Guide, AI Concierge and on-site Traveller Hub.

### 0:22–0:42 — The Build Week extension

**Screen:** Scroll from the trajectory to Mission Control.

**Narration:**

> That live foundation existed before Build Week. The new work is Atlas Mission Control: a closed operating loop that turns one traveler request into both a grounded journey and a structured signal telling a small tourism operator what to improve next.

### 0:42–1:25 — Working guest journey

**Screen:** Select **Rainy family**, click **Generate Atlas mission**, then show the first day and its verification labels.

**Narration:**

> This family has two rainy days, no car and needs short, flexible transfers. Atlas interprets audience, mobility, pace and weather sensitivity. A deterministic guardrail engine ranks the destination knowledge first, so the mission remains coherent and car-free. Every recommendation resolves to an owned Atlas source. Opening status, access and timetables are treated as volatile and visibly marked for a live check instead of being guessed.

### 1:25–1:58 — Operator value

**Screen:** Open **Mission control** and move across demand signals, trust layer and opportunities.

**Narration:**

> The same request becomes anonymous operational intelligence: family suitability, car-free continuity and weather-resilient demand. Atlas identifies trust risks and prioritizes the next partner, content or conversion improvement. No traveler profile is stored. One interaction improves the guest journey and the operator’s next decision.

### 1:58–2:18 — Evidence contract

**Screen:** Open **Evidence** and one Atlas source in a new tab.

**Narration:**

> The evidence view makes the trust contract inspectable. Judges can follow each source into the live Atlas network, while the application distinguishes stable local knowledge from facts requiring verification.

### 2:18–2:42 — Codex and GPT-5.6

**Screen:** Show `src/mission-service.mjs`, briefly highlight `model: gpt-5.6`, `store: false` and the strict JSON schema; then show the passing tests.

**Narration:**

> Codex helped us design and build the architecture, ranking engine, API, interface, tests, safeguards and submission documentation. GPT-5.6 receives only the already-grounded mission and returns strict structured output. It can improve interpretation and operator insight, but it cannot add places or mutable facts. If the API is unavailable, the deterministic product still works.

### 2:42–2:55 — Winning close

**Screen:** Return to the hero or use the final product view.

**Narration:**

> Atlas turns independent hospitality knowledge into a destination operating system: better journeys, more local value and a self-improving regional knowledge loop. Not another travel answer—grounded action and learning.
